# Structure First (Level 12: Meta-Conservation Law)
## Starlette routing.py Analysis

---

## The Falsifiable Claim

**This code's deepest structural problem is that it conflates routing decisions with authorization decisions through the `Match` enum's three-valued logic, forcing a single `PARTIAL` result to simultaneously mean "path matched, method failed" (for 405 responses), "path matched, defer to deeper matching" (for Mount nesting), and "path matched, keep searching" (for route ordering) — semantic overloading that makes correct HTTP behavior impossible.**

---

## Three Experts Test the Claim

### Expert 1: DEFENSE (The claim is correct)

The defense points to concrete evidence:

**Lines 163-168 (Route.matches):**
```python
if self.methods and scope["method"] not in self.methods:
    return Match.PARTIAL, child_scope
else:
    return Match.FULL, child_scope
```
A `PARTIAL` match captures `child_scope` with `endpoint` populated. This endpoint is *the wrong endpoint* for the method — it's the class/function registered for allowed methods, not the actual incoming method.

**Lines 263-272 (Router.app):**
```python
partial = None
for route in self.routes:
    match, child_scope = route.matches(scope)
    if match is Match.FULL:
        scope.update(child_scope)
        await route.handle(scope, receive, send)
        return
    elif match is Match.PARTIAL and partial is None:
        partial = route
        partial_scope = child_scope
```
Only the *first* `PARTIAL` is captured. If `/users` is registered for both `GET` (Route A) and `POST` (Route B), and a `DELETE /users` arrives:
- Route A returns `PARTIAL` (path matches, method doesn't)
- Route B returns `PARTIAL` (path matches, method doesn't)
- Only Route A's scope is kept

The 405 response from `partial.handle()` can only know about Route A's methods. HTTP compliance requires `Allow: GET, POST` — **structurally impossible**.

### Expert 2: ATTACK (The claim is wrong or overstated)

The attack identifies three weaknesses:

1. **Conflation isn't the problem — it's a solution.** The `PARTIAL` state exists precisely to distinguish "no match" from "method not allowed." Without it, you'd need separate path-matching and method-matching phases, which is either slower (match all paths first) or incomplete (first-match-wins for both).

2. **HTTP 405 compliance is niche.** Most applications don't need correct `Allow` headers. The current behavior (return 405 with whatever methods the first matching route supports) is "good enough" for the 99% use case.

3. **The real problem is mutation.** Lines 169 and 265 do `scope.update(child_scope)`. This mutates the incoming scope *in place*. If a middleware retries routing after a failed match, the scope is corrupted. The `Match` enum isn't the culprit — **scope mutation** is.

**The attack's alternative claim:** The deepest problem is *not* Match semantic overloading, but rather **imperative scope mutation that makes routing stateful and non-reentrant**.

### Expert 3: PROBE (What do both take for granted?)

The probe finds the shared assumption:

Both experts assume **a single routing decision must be made per request**. The defense wants that decision to be "correct" (all applicable methods). The attack wants that decision to be "safe" (no mutation).

What if the assumption is wrong? What if routing should produce *multiple* candidate matches, not a single decision?

The probe also notices both miss:

**The `Mount` case (lines 197-218):** `Mount.matches` returns `Match.FULL` when the path prefix matches, regardless of whether the child router has any matching routes. A Mount never returns `PARTIAL`. This means:
- `/api/users` with Mount at `/api` → `FULL` match, defer to child
- Child Router has no `/users` route → 404
- But the correct response might be 405 if `/api/users` exists for other methods

**The semantic overloading of `FULL`:** For Route, `FULL` means "path and method match." For Mount, `FULL` means "path prefix matches." The same enum value means different things for different subclasses.

---

## The Gap (Diagnostic)

The dialectic revealed:
- Defense: Correct about 405 impossibility, wrong about root cause (not conflation, but *first-match-wins*)
- Attack: Correct about mutation, wrong about severity (mutation is fixable; first-match-wins is structural)
- Probe: Correct that both assume single-decision routing, revealed Mount semantic inconsistency

**The transformed claim:** The deepest problem is that **first-match-wins semantics over a heterogeneous route list makes complete HTTP method information structurally inaccessible**. Routes and Mounts return the same `Match` values with different semantics, and the Router processes them in order without distinguishing type, making it impossible to aggregate method information across multiple routes that share a path.

---

## The Concealment Mechanism

**Name: Semantic Boundary Smoothing**

The code *conceals* the Route/Mount semantic mismatch by:
1. Making both inherit from `BaseRoute` with identical method signatures
2. Using the same `Match` enum for both
3. Processing both in a single loop without type discrimination

This is "smooth" because it looks like clean polymorphism. But the *behavior* of `Match.FULL` is fundamentally different:
- Route.FULL: "This request should be handled by this endpoint"
- Mount.FULL: "This request should be handled by my child router"

The smoothness *hides* that these are two different operations (termination vs. delegation) encoded as one.

---

## Application: Engineer an Improvement That Deepens Concealment

**Goal:** Make the code "better" (pass code review) while making the real problem harder to see.

```python
class Match(Enum):
    NONE = 0
    PARTIAL = 1  
    FULL = 2
    
    def __or__(self, other):
        """Combine matches: the highest non-NONE wins."""
        if self == Match.NONE:
            return other
        if other == Match.NONE:
            return self
        return Match.FULL if self == other == Match.FULL else Match.PARTIAL
```

**Code review would approve this because:**
- Adds useful functionality (match composition)
- Has documentation
- Seems "principled" (composable operations)

**What it conceals:** It makes `Match` *look* even more like a unified semantic type by adding algebraic operations. But composing a Route.PARTIAL with a Mount.FULL still has undefined semantics. The operation *deepens* the concealment by adding mathematical structure that implies coherence where there is none.

---

## Three Properties Visible Because of the Improvement

1. **Match values are now context-dependent in a new way.** The `|` operator assumes matches are commensurable, but `(Route.PARTIAL | Mount.FULL)` produces `FULL`, which suggests the Mount should handle the request — even though the Route's PARTIAL meant "method not allowed" and the Mount's FULL meant "prefix matched." The composition *inverts* the Route's veto.

2. **The aggregation problem becomes operator-shaped.** If we wanted to collect all allowed methods, we'd need something like `Match.PARTIAL.allowed_methods | other.PARTIAL.allowed_methods`. But `Match` has no such field. The improvement *reveals* that Match is missing the data needed for its own composition to be meaningful.

3. **Mount never returns PARTIAL, so Mount | anything = anything.** The improvement makes this *visible* because you'd expect the operator to be useful, but it's not useful for Mounts. The code review would say "cool feature" without noticing it's only meaningful for Routes.

---

## Apply the Diagnostic to the Improvement

The improvement (`Match.__or__`) **conceals that Match values are not actually composable** because they carry implicit context (what kind of route produced them) that the operator ignores.

**What property of the original problem is visible only because the improvement recreates it?**

The original problem (first-match-wins loses method information) becomes visible in a new form: **composition loses method information**. The `|` operator picks a winner (FULL > PARTIAL > NONE) rather than aggregating data. The improvement *recreates* the information-loss pattern at the algebraic level.

## Second Improvement: Address the Recreated Property

**Goal:** Fix the "composition loses method information" problem that the first improvement exposed.

```python
class Match(Enum):
    NONE = 0
    PARTIAL = 1  
    FULL = 2
    
    def __or__(self, other):
        """Combine matches, aggregating method information."""
        if self == Match.NONE:
            return other
        if other == Match.NONE:
            return self
        # Create a composite match that carries both results
        return CompositeMatch(self, other)

class CompositeMatch:
    """A match that combines multiple route matches."""
    def __init__(self, *matches):
        self.matches = matches
        self._methods = set()
        for m in matches:
            if hasattr(m, 'methods'):
                self._methods.update(m.methods)
    
    @property
    def methods(self):
        return self._methods
```

**Code review would approve this because:**
- Actually addresses the information-loss problem
- Provides a way to get all allowed methods
- Uses composition pattern

---

## Apply the Diagnostic Again

**What does the second improvement conceal?**

It conceals that **`Match` is no longer an enum**. By introducing `CompositeMatch`, we've broken the assumption that `match is Match.FULL` or `match == Match.PARTIAL`. Any code that does `if match is Match.FULL:` will now fail for composite matches.

**Where does this break?**

Line 264:
```python
if match is Match.FULL:
```
Now fails for `CompositeMatch`. We'd need to add:
```python
if match == Match.FULL or (isinstance(match, CompositeMatch) and match.highest() == Match.FULL):
```

**The structural invariant:** We keep *adding* information to `Match` to fix specific problems, but every addition breaks code that assumed `Match` was simple.

---

## The Structural Invariant

**Name: The Complexity Ratchet**

Every improvement to the Match/Route/Mount system adds a new concept (operator, composite, method set) that:
1. Solves the immediate problem
2. Breaks code that relied on the previous simplicity
3. Requires more code to handle the new case
4. Creates new edge cases that require further improvement

The invariant: **Match's simplicity and Match's completeness are inversely related.** You cannot have both "easy to pattern-match on" and "carries all routing information."

---

## Invert the Invariant

**Design where the impossible property (completeness + simplicity) becomes trivially satisfiable:**

```python
# Inverted design: routing produces ALL matches, consumer decides
@dataclass
class RouteMatch:
    route: BaseRoute
    path_params: dict
    method_allowed: bool
    allowed_methods: set

class Router:
    def all_matches(self, scope) -> list[RouteMatch]:
        """Return ALL matches, sorted by specificity."""
        matches = []
        for route in self.routes:
            match = route.matches(scope)
            matches.append(match)
        return sorted(matches, key=lambda m: m.specificity, reverse=True)
    
    async def __call__(self, scope, receive, send):
        matches = self.all_matches(scope)
        
        # Consumer decides: first FULL match, or aggregate methods from PARTIALs?
        full_matches = [m for m in matches if m.method_allowed]
        if full_matches:
            await full_matches[0].handle(scope, receive, send)
            return
        
        partial_matches = [m for m in matches if m.route.path_matches(scope)]
        if partial_matches:
            all_methods = set()
            for m in partial_matches:
                all_methods.update(m.allowed_methods)
            response = MethodNotAllowedResponse(allow=all_methods)
            await response(scope, receive, send)
            return
        
        await self.default(scope, receive, send)
```

**What's trivially satisfiable now:**
- Getting all allowed methods: `all_methods = set().union(*[m.allowed_methods for m in partial_matches])`
- Match simplicity: `RouteMatch` is a dataclass with no enum tricks
- Completeness: every route that matches the path contributes to the result

---

## The New Impossibility

The inverted design creates: **Performance is now impossible.**

```python
def all_matches(self, scope) -> list[RouteMatch]:
    for route in self.routes:  # MUST check ALL routes
        match = route.matches(scope)
        matches.append(match)
```

The first-match-wins design stops at the first `FULL` match. The inverted design *must* check all routes to build the complete method set. For a router with 1000 routes and a request matching route 2, we do 998 unnecessary checks.

**The conservation law between original and inverted impossibilities:**

**Completeness × Performance = Constant**

- Original: High performance (early exit), low completeness (missing method info)
- Inverted: High completeness (all methods), low performance (no early exit)

You cannot have both correct 405 responses *and* O(1) routing in the worst case.

---

## Apply the Diagnostic to the Conservation Law

**What does "Completeness × Performance = Constant" conceal about this specific problem?**

It conceals that **the trade-off only exists because routes are stored as a list and checked sequentially.** 

If routes were indexed by path pattern (e.g., in a trie), you could:
1. Find all routes matching `/users` in O(log n) 
2. Aggregate their methods
3. Return 405 with complete `Allow` header

The conservation law assumes a *specific data structure* (flat list) and presents its limitations as fundamental.

**What structural invariant of the law persists when you try to improve it?**

If I say "use a trie to avoid the trade-off," the law shifts to: **Index Size × Insertion Speed = Constant.**

The trie would be larger (more nodes) and slower to build (more pointer chasing during insertion). The trade-off *migrates* from lookup to insertion.

---

## The Meta-Law (Conservation Law of the Conservation Law)

**Name: Trade-off Migration**

When you "solve" a conservation law by changing the design, the trade-off doesn't disappear — it **migrates** to a different dimension of the same system.

For Starlette routing:
- Flat list: Lookup × Completeness = constant
- Trie: Index Size × Insertion Speed = constant  
- Hash table: Memory × Collision Handling = constant

**The meta-law:** For any routing data structure, there exists a dimension where the trade-off reappears. This is not a general principle about "all trade-offs" — it's a specific prediction about **this routing problem**.

**Concrete, testable consequence:** If Starlette were rewritten to use a radix trie for O(log n) path lookup, the new impossibility would be: "Adding a route with many path parameters would be O(n) in the number of parameters, because each parameter creates trie branch points." The trade-off migrates from *lookup* to *route registration*.

## Summary

**Deepest structural problem:** First-match-wins over heterogeneous routes makes complete HTTP method information inaccessible.

**Conservation law:** Completeness × Performance = Constant (for flat-list routing)

**Meta-law:** Trade-off Migration — solving a conservation law migrates the trade-off to another dimension (for this routing system: from lookup to registration)

**Critical fixable bug:** Nested `app` function in `request_response` (lines 24-30) shadows outer scope and breaks exception handling.

**Structural limitation:** 405 responses cannot include all allowed methods without checking all routes.
