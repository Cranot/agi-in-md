# Structural Analysis: Starlette Routing

## Three Properties Claimed

1. **Arbitrary Nesting** — `Mount` composes sub-apps to any depth; each Mount can contain routes or other Mounts
2. **Parameter Isolation** — Each route level extracts its own `path_params` without collision
3. **Scope Continuity** — A single mutable `scope` dict flows through the entire dispatch chain

## Proof of Impossibility

The trilemma lives in `Mount.matches()` and `BaseRoute.__call__()`:

```python
# Mount.matches() — line ~230
path_params = dict(scope.get("path_params", {}))
path_params.update(matched_params)  # ← MERGE with overwrite
child_scope = {"path_params": path_params, ...}

# BaseRoute.__call__() — line ~115
scope.update(child_scope)  # ← MUTATE in place
```

**The collision:**

If parent Mount has `/{tenant_id}/dashboard` and child route has `/{user_id}/profile`, parameters coexist. But if **any two levels** declare the same parameter name (`/{id}`), the child's value **silently overwrites** the parent's.

Property 2 (isolation) requires parameter namespacing. Property 1 (arbitrary nesting) means sub-apps are developed independently—they don't coordinate parameter names. Property 3 (scope continuity) provides no namespacing mechanism—just a flat dict with `.update()`.

You cannot have all three:
- **Arbitrary nesting** + **Scope continuity** → collisions when parameter names repeat
- **Parameter isolation** + **Scope continuity** → requires global parameter name registry (breaks arbitrary nesting)
- **Arbitrary nesting** + **Parameter isolation** → requires scope copying/namespacing (breaks continuity, adds allocation cost)

## Conservation Law

**Composition depth × Parameter namespace isolation = constant**

The deeper the routing tree, the higher the probability of silent parameter collision. At depth 1 with controlled routes: safe. At depth 4 with third-party sub-apps: parameter names become a coordination problem with no enforcement mechanism.

---

**What was actually sacrificed:** Parameter isolation. The design allows arbitrary nesting and scope continuity, but parameter collision is a *runtime* failure mode, not a compile-time check. Sub-apps must coordinate parameter names by convention, not by structure.
