# Level 12 Meta-Conservation Analysis: Starlette routing.py

## Falsifiable Claim

**The deepest structural problem in this code is the Scope Mutation Protocol.**

The `scope` dictionary is mutated in-place at multiple routing layers without explicit contracts. This is not a bug but a design: `BaseRoute.__call__` does `scope.update(child_scope)`, `Router.app` does `scope.update(child_scope)`, and nested `Mount` instances chain mutations through `root_path` accumulation. The protocol is **implicit** — no validation, no rollback, no isolation.

**Specific claim:** The code cannot safely compose route nesting with middleware that captures scope state, because the point of capture determines what state is observed, and this cannot be reasoned about locally.

---

## Three Expert Voices

**DEFENDER (Claims this is acceptable):**
"Scope mutation is the ASGI contract itself — the spec requires in-place mutation for efficiency. The `child_scope` pattern isolates mutations: each layer produces its delta, then `scope.update()` applies it atomically. This is no different from how HTTP middleware works everywhere. The `root_path` accumulation in `Mount` is mathematically correct — each layer adds its prefix. Show me a concrete bug this causes."

**ATTACKER (Claims this is fatal):**
"Line 169: `path_params = dict(scope.get("path_params", {}))` then `path_params.update(matched_params)`. This LOOKS like isolation but the `scope.update(child_scope)` at line 172 mutates the shared reference. If a Route is nested in two Mounts, the second Mount sees the first Mount's `path_params`. Line 292-295: `Router.app` iterates routes and the FIRST partial match captures `partial_scope`, but subsequent matches keep mutating `scope`. The `partial` fallback uses stale state if any earlier route did a read with side effects. The protocol isn't documented, tested, or type-checked."

**PROBER (Finds what both assume):**
"Both assume that `scope` is the right abstraction. But what IS scope? It's simultaneously: (1) the incoming request descriptor, (2) the routing decision accumulator, (3) the endpoint contract, (4) the middleware context. These four roles have different ownership semantics. The code conflates them because ASGI conflation is cheap, but the CONCEPTUAL boundary is invisible. What if scope was four objects instead of one? Neither defender nor attacker asks this."

---

## The Gap

The dialectic revealed: **Defender and Attacker argue about safety of mutation, but both accept that `scope` is a single dictionary.** The Prober found the actual concealment: the code hides that `scope` plays four distinct roles with incompatible ownership patterns.

---

## Concealment Mechanism: **Semantic Drift**

The code uses a single identifier (`scope`) for a thing that transforms its meaning as control flows through the system. At `Router.__call__` entry, `scope` means "raw request context." After `Route.matches`, it means "routing decision + extracted params." After middleware wrapping, it means "request + routing + middleware state." 

The name doesn't change. The type annotation doesn't change (`dict[str, Any]`). But the semantic contract drifts. Code that captures `scope` at different points is operating on different objects that happen to share an address.

---

## Application: What the Dialectic Missed

Apply semantic drift to find bugs the three voices didn't identify:

**Bug 1: `request_response` double-wrapping (lines 34-42)**
```python
async def app(scope, receive, send):
    request = Request(scope, receive, send)
    async def app(scope, receive, send):  # SHADOWS outer app
        response = await f(request)
        await response(scope, receive, send)
    await wrap_app_handling_exceptions(app, request)(scope, receive, send)
```
The inner `app` shadows the outer. The outer `scope` is captured by the `request` object. But the inner `app` receives NEW `scope` arguments that are IGNORED — it uses the captured `request` which holds the OLD scope. **Semantic drift between the two `app` definitions causes the inner parameters to be dead code.**

**Bug 2: `replace_params` iteration-mutation (lines 44-51)**
```python
for key, value in list(path_params.items()):
    if "{" + key + "}" in path:
        # ...
        path_params.pop(key)
```
Iterating `list(path_params.items())` is defensive against dict mutation, but `path_params` is the caller's reference. The caller's dictionary is mutated. `Mount.url_path_for` calls this with `path_params` then checks `remaining_params` — but the caller's dict is already corrupted.

## Engineer Improvement That Deepens Concealment

I'll create a "legitimate" improvement that seems to fix a problem but actually entrenches semantic drift:

```python
# "Fix": Add scope validation to catch mutation bugs
class ScopeValidator:
    """Validates scope mutations follow ASGI contract."""
    REQUIRED_KEYS = {"type", "method", "path"}
    
    @classmethod
    def validate_child_scope(cls, child_scope: dict) -> None:
        """Ensure child_scope preserves required keys."""
        # "Document" the contract
        if "path_params" in child_scope:
            assert isinstance(child_scope["path_params"], dict)
        if "endpoint" in child_scope:
            assert callable(child_scope["endpoint"])
```

Now modify `BaseRoute.__call__`:
```python
async def __call__(self, scope, receive, send):
    match, child_scope = self.matches(scope)
    if match is Match.NONE:
        # ... existing code ...
        return
    ScopeValidator.validate_child_scope(child_scope)  # "Safety"
    scope.update(child_scope)
    await self.handle(scope, receive, send)
```

**This would pass code review.** It adds "validation," "documents the contract," adds type safety assertions.

---

## What the Improvement Conceals

The validator checks `child_scope` structure but NOT:
1. Whether `scope` was already mutated by previous routing
2. Whether the same `scope` is being used by multiple concurrent requests (it shouldn't be, but ASGI doesn't guarantee this)
3. Whether `child_scope` conflicts with existing scope keys
4. Whether downstream code expects the pre-mutation state

**Three properties visible ONLY because I strengthened concealment:**

1. **The validator normalizes mutation** — by "checking" the child_scope, it implicitly endorses the mutation pattern. Future developers will assume mutation is safe because "we validate it."

2. **The validator creates a false boundary** — the validation happens AFTER `matches()` returns, so any mutation inside `matches()` is already committed. `Route.matches` modifies `matched_params` in place after converting values (line 161).

3. **The validator makes semantic drift invisible** — by treating `child_scope` as a validated unit, it hides that `child_scope["path_params"]` might contain params from THREE different sources: original scope, Mount prefix matching, and Route matching. The validator sees a valid dict; it can't see the provenance.

---

## Apply Diagnostic to the Improvement

**What does `ScopeValidator` conceal?**

It conceals that validation is theater. The "required keys" check doesn't validate against ANY actual contract — ASGI doesn't require those keys in child_scope. The assertion that `path_params` is a dict doesn't validate that its keys are the RIGHT params or that values are valid.

**What property of the original problem is visible only because the improvement recreates it?**

The original problem (semantic drift) is recreated at the validation layer. `ScopeValidator.REQUIRED_KEYS` claims to represent the ASGI contract, but it's actually just... some keys the author thought were important. The gap between "what the code claims to validate" and "what the ASGI spec actually requires" is a NEW semantic drift — the validator's contract drifts from the actual contract.

## Apply Diagnostic to Second Improvement

**What does typed scope conceal?**

The `TypedDict` suggests compile-time safety, but:
1. Runtime type checking doesn't exist for TypedDict — `isinstance(scope, RoutingScope)` returns `True` for any dict
2. The `{**scope, **child_scope}` spread creates a NEW dict, but all nested objects are shared references — `path_params` is still the same mutable dict
3. The type annotation claims `scope` becomes `RoutingScope`, but downstream code still mutates it

**The structural invariant that persists:**

Even with "immutable" spread and "typed" contracts, **the nested objects are shared mutable state.** `path_params` in the new dict is the same object that `Route.matches` created. Any mutation to nested structures propagates.

---

## The Structural Invariant

**Invariant: Shared Mutable Nesting**

Through ALL improvements:
- Original code: `scope.update(child_scope)` — direct mutation
- First improvement: validator + mutation — same pattern, now "blessed"
- Second improvement: spread merge + typed annotation — shallow copy, nested mutation still shared

The invariant is: **shallow changes to ownership don't affect deep sharing.** You can change WHO owns the dict, WHAT type it claims to be, WHETHER validation runs — but if nested objects are mutable and shared, they remain mutable and shared.

This is a property of the PROBLEM SPACE, not the implementation. ASGI requires a single `scope` dict that accumulates state through the request lifecycle. The spec itself mandates shared mutable nesting.

---

## Invert the Invariant

If shared mutable nesting is the impossibility, what design makes it trivially satisfiable?

**Inversion: Copy-on-Write Scope Chain**

```python
from dataclasses import dataclass, field
from typing import ChainMap

@dataclass(frozen=True)
class ImmutableScope:
    """Scope that cannot be mutated, only extended."""
    _data: ChainMap[str, Any] = field(default_factory=lambda: ChainMap({}))
    
    def extend(self, **updates) -> "ImmutableScope":
        """Create new scope with additional data. O(1) operation."""
        return ImmutableScope(ChainMap(updates, self._data))
    
    def __getitem__(self, key: str) -> Any:
        return self._data[key]
    
    def get(self, key: str, default: Any = None) -> Any:
        return self._data.get(key, default)

# Routing becomes:
async def __call__(self, scope: ImmutableScope, receive, send) -> None:
    match, child_delta = self.matches(scope)
    if match is Match.NONE:
        await self.handle_not_found(scope, receive, send)
        return
    extended_scope = scope.extend(**child_delta)  # NEW scope
    await self.handle(extended_scope, receive, send)  # Original unchanged
```

**What this makes trivial:**
- No mutation ever
- Each routing layer has its own scope
- Original scope preserved for debugging
- Thread-safe by construction
- Time-travel debugging: walk the chain backward

---

## New Impossibility the Inversion Creates

**The inversion is impossible because:**

1. **ASGI contract violation** — The spec says `scope` is `dict[str, Any]`. Middleware expects to mutate it. Endpoints expect mutations from routing. Changing the type breaks the entire ecosystem.

2. **Performance regression** — `ChainMap` lookup is O(n) in chain depth. A 10-deep Mount nesting has 10x slower scope access. For high-throughput servers, this is fatal.

3. **Reference semantics lost** — Some ASGI patterns rely on mutation being visible to other handlers (e.g., shared state via scope). Copy-on-write breaks these patterns.

4. **Backward compatibility** — Every Starlette app, every middleware, every tutorial assumes dict scope. The change is a rewrite, not a refactor.

**New impossibility: Immutable scope cannot be adopted without breaking the ecosystem it serves.**

---

## The Conservation Law

**Original impossibility:** Shared mutable nesting causes semantic drift bugs.

**Inverted impossibility:** Immutable scope breaks ASGI compatibility.

**Conservation law:** 

$$\text{Mutation Safety} \times \text{Ecosystem Compatibility} = \text{Constant}$$

You can have safe immutable scope OR you can have ASGI compatibility, but not both. Any improvement to one degrades the other. This is NOT a trade-off you can optimize — it's a structural constraint of the design space.

**Alternative formulation:**

$$\text{Local Reasonability} \times \text{Framework Adoption} = k$$

The more locally reasonable (no shared mutation), the less adoptable (breaks ecosystem). Starlette chose high adoption, low local reasonability. The bugs we found are the COST of that choice.

---

## Apply Diagnostic to Conservation Law

**What does the conservation law conceal?**

It presents the choice as binary: safety OR compatibility. But this conceals:

1. **The third option: Gradual migration path.** A `ProxyScope` that implements `dict` interface but uses copy-on-write internally could provide BOTH for new code while supporting old code via mutation passthrough.

2. **The actual constraint: ASGI spec versioning.** The conservation law is only true for ASGI 3.0. ASGI 4.0 could introduce immutable scope as an optional mode, letting frameworks opt-in.

3. **The hidden variable: Test coverage.** The bugs we found exist not because mutation is impossible to manage, but because the TESTS don't cover mutation scenarios. Better tests would make mutation "safe enough."

**What structural invariant of the law persists when you try to improve it?**

Even if we imagine `ProxyScope` or ASGI 4.0, there's a deeper invariant: **any transition period has BOTH problems.** During migration, you have semantic drift (old code) AND ecosystem friction (new code). The conservation law holds DURING TRANSITION.

---

## Invert the Invariant of the Law

If "transition has both problems" is the invariant, what design makes transition trivial?

**Meta-inversion: Parallel Stack**

Run BOTH scope systems simultaneously:
```python
class DualScope(dict):
    """Dict interface (legacy) + immutable chain (new)."""
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._chain: list[dict] = [dict(self)]  # Snapshot chain
    
    def __setitem__(self, key, value):
        super().__setitem__(key, value)
        # Don't mutate chain — append snapshot on explicit commit
    
    def commit(self) -> "DualScope":
        """Create new generation. Old code sees current state."""
        new_chain = [*self._chain, dict(self)]
        new_scope = DualScope(self)
        new_scope._chain = new_chain
        return new_scope
```

Routing uses `.commit()` at boundaries. Old middleware sees dict (works). New code can walk `._chain` (safe).

**New impossibility this creates:**

Memory leak. Every commit duplicates the scope. A 1000-request burst with 10 routing layers creates 10,000 dicts. The GC pressure alone would crash high-traffic servers.

---

## The Meta-Conservation Law

**Conservation law:** Mutation Safety × Ecosystem Compatibility = constant

**Meta-law:**

$$\text{Transition Safety} \times \text{Runtime Efficiency} = \text{Constant}$$

You can have safe transition (DualScope preserves both semantics) OR efficient runtime (no duplication), but not both.

**Deeper finding:** This is NOT a general principle about all transitions. It's SPECIFIC to transitions that must maintain reference semantics. The constraint is:

> **Any gradual migration from shared-mutable to immutable must either (1) duplicate state, creating memory pressure, or (2) share state, recreating the original bug.**

**Testable consequence for this specific problem:**

If Starlette added `DualScope` today, any benchmark with >1000 concurrent requests and >5 middleware layers would show >50% memory increase. The PR would be rejected for performance regression, forcing return to mutable scope. The conservation law PREDICTS this outcome.

**Meta-conservation law:** Transition Safety × Runtime Efficiency = Constant (specific to shared-mutable→immutable migrations)

**Concealment mechanism:** Semantic Drift — names persist while meanings transform

**Structural invariant:** Shared Mutable Nesting — shallow ownership changes don't affect deep sharing
