# Structural Analysis: Starlette routing.py

## Deepest Structural Problem

**Claim:** The routing system is built on a **progressive disclosure fallacy** — it pretends that partial match information exists to enable intelligent routing decisions, but the partial match state is discarded before it can inform anything except a binary "use this route or don't."

The `Match` enum (NONE/PARTIAL/FULL) suggests a tri-state matching system. But examine `Router.app()`:

```python
partial = None
for route in self.routes:
    match, child_scope = route.matches(scope)
    if match is Match.FULL:
        # ... executes immediately and returns
    elif match is Match.PARTIAL and partial is None:
        partial = route  # stores ONLY the first partial
```

**First partial wins, all others vanish.** The partial match exists not to inform, but to enable one specific behavior (405 Method Not Allowed via `partial.handle()`). The tri-state is theatrical — it's actually binary with a costume change.

---

## Three Experts Test

**Defender:** "This is correct design. HTTP semantics require 405 when path matches but method doesn't. PARTIAL exists precisely for this. First-come-first-served is the documented behavior for route ordering."

**Attacker:** "The defender confuses implementation with semantics. Yes, 405 needs *some* signal, but why discard information? What if I want to know ALL routes that partially matched? What if I want to route based on partial match patterns? The design forecloses these without acknowledging it chose a path."

**Probing Assumptions:** "What makes you believe partial matches are rare enough to store only one? What if a pathological route configuration has 50 partial matches — was storing 49 scopes really the bottleneck? Or did no one ask what partial matches *mean* as aggregate data?"

**Verdict:** The defender is right about HTTP semantics but wrong about the sufficiency of the design. The real problem is **information laundering** — partial match state exists long enough to serve one purpose, then vanishes as if it never existed.

---

## Concealment Mechanism

**Name:** **Scoped Amnesia** — The system creates information (child_scope from partial matches), uses it once for a single decision path, then allows it to fall out of scope. The information was real, its disappearance is unmarked, and no trace remains that it ever existed.

This conceals:
1. **Route overlap patterns** — you cannot query "which routes would have matched if I changed X?"
2. **Configuration drift** — routes that shadow each other non-obviously
3. **Debugging provenance** — when 405 fires, you know *one* partial matched, but not how many tried

---

## Improvement That Deepens Concealment

**Engineering:**

```python
class Router:
    def __init__(self, routes=None, ..., match_strategy="first_full"):
        self.match_strategy = match_strategy  # "first_full" | "all_partials"
```

Add a `match_strategy` parameter. Default `"first_full"` preserves existing behavior. New `"all_partials"` accumulates all partial scopes into a list attached to the request for introspection.

**Three Properties Visible Only Through This Strengthening:**

1. **Route shadow depth** — count of partials reveals how many routes a request "almost" matched
2. **Configuration coherence** — partial match clusters reveal whether routes are genuinely distinct or accidentally overlapping
3. **Migration safety** — before removing a route, query what would have partially matched to its consumers

---

## Diagnostic Applied to Improvement

**What does this improvement conceal?**

The `match_strategy` parameter **reifies a false choice**. By making "collect all partials" an option, it suggests the default (discard) is a valid design choice rather than an unexamined inheritance. The improvement conceals that **partial matches were never meant to be data** — they were an implementation detail of 405 handling that accidentally became API.

More deeply: the parameter suggests "you can have it either way." But the meaningful question isn't "collect or discard?" — it's "why does partial match exist as a concept at all?" The improvement prevents this question by answering a shallower one.

---

## Second Improvement

**Engineering:**

Eliminate `Match.PARTIAL` entirely. Replace with:

```python
class Match(Enum):
    NONE = 0
    FULL = 1

class Route:
    def matches(self, scope):
        # Returns (Match, child_scope, method_allowed: bool)
```

The method-check moves out of `matches()` into a separate concern. Routes match paths, not paths+methods. Method validation becomes middleware or a `MethodValidator` wrapper.

**Effect:** Partial match was never a routing concept — it was HTTP semantics leaking into the matching layer. By extracting it, the routing system becomes pure (matches paths) and HTTP concerns live elsewhere.

---

## Structural Invariant

**Name:** **The Responsibility Boundary Must Blur**

Starlette's routing cannot stay pure because ASGI conflates transport (path routing) with semantics (method handling). The system is forced to know about HTTP methods to return correct status codes. The invariant:

> **Routing precision × protocol awareness = constant**

The more precisely you match routes, the more you must know about protocol semantics. Pure path routing would require external protocol handlers — but those handlers need routing information to make decisions, creating a circular dependency.

---

## Inverted Invariant

**Invert:** Routing precision × protocol awareness = constant

→ **Routing imprecision × protocol ignorance = constant**

**New Impossibility:** You cannot build a routing system that is simultaneously:
- Agnostic to the protocol being routed (HTTP, WebSocket, custom)
- Precise enough to return protocol-appropriate errors (405 vs 404)

The inverse is equally impossible: a routing system that knows nothing about protocols cannot generate protocol-correct responses. It would route WebSocket connections perfectly but return HTTP 404 text to them.

---

## Conservation Law

**The Finding:**

> **Routing abstraction × Error specificity = constant**

The more abstract (protocol-agnostic) the routing layer, the less specific its error responses can be. Starlette chose mid-abstraction: it knows just enough HTTP to return 405, but not enough to explain *why* 405 (which methods *are* allowed?). The conservation law explains why `Match.PARTIAL` exists but stores minimal information — storing more would require deeper HTTP coupling, violating the balance.

---

## Meta-Conservation Law (Applied to This Analysis)

**Apply diagnostic to my own law:**

My conservation law (Routing abstraction × Error specificity = constant) conceals the **third term**: developer ergonomics. Starlette didn't arrive at its balance through abstract reasoning — it arrived through ergonomic pressure. Developers want:
- Simple route definitions (`@app.route("/users/{id}")`)
- Correct error codes automatically
- No explicit error handlers for common cases

The conservation law is real but **incomplete**. The full equation:

> **Routing abstraction × Error specificity × Developer convenience ≈ constant**

You can optimize any two at the expense of the third.

**Meta-law:**

> **Every conservation law in software is a projection of stakeholder priorities onto technical dimensions.** The "constants" aren't mathematical — they're social settlements. Starlette's balance reflects what its contributors valued, not what's objectively optimal.

## Prediction

If Starlette adds gRPC or HTTP/3 support, the `Match` system will either:
1. Grow a fourth state (`PARTIAL_PROTOCOL`?), creating exponential complexity
2. Be replaced entirely with a capability-matching system

The conservation law predicts option 2 — but the social settlement (contributor familiarity with current code) predicts option 1.
